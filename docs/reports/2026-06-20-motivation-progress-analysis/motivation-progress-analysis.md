# Motivation Progress Analysis

## Summary
Motivation progress should be generated from a small set of household contribution events, not from every household activity and not from a currency ledger. The best direction is a goal-scoped contribution model: a Motivation goal declares which kinds of contribution count, and completed or parent-recognized household actions advance only matching active goals.

Recommended direction:
- Task completion can contribute, but not every completed Task should automatically advance Motivation.
- Family and individual goals should choose eligible contribution sources by intent, such as selected tasks, shared tasks, routines, or parent-recognized helpful moments.
- Helpful Moments should become a future parent-entered recognition source for initiative, cooperation, and support that Tasks cannot represent well.
- Parent recognition should be supported as a manual progress source, but bounded so Motivation does not become constant adult bookkeeping.
- Motivation progress should use child-readable units such as steps, stars, checkmarks, days, or contributions, while parents see clearer source and eligibility labels.
- Reward Economy should remain separate. Motivation progress says, “we are growing as helpers”; Reward Economy later may say, “this approved event changed a balance.”

The core product principle is: **Motivation recognizes contribution patterns; Tasks execute responsibilities; Reward Economy may later account for exchangeable value.**

## Motivation Data Flow
Motivation progress should be generated by contribution events that represent meaningful participation in household life.

Sources that belong:
- **Selected Task completion** when the task expresses an age-appropriate responsibility or a goal-relevant routine.
- **Shared task completion** when the family goal is about cooperation, everyone helping, or household reset work.
- **Family goal participation** when the desired outcome is inclusive participation rather than raw task count.
- **Helpful Moments** for non-task contributions such as helping a sibling, noticing a need, or taking initiative.
- **Parent recognition** when a caregiver deliberately credits a child or the family for meaningful effort that the task board did not capture.
- **Routine completion** when Tasks later support recurring routines or when a goal uses a routine-like source.

Sources that should not automatically belong:
- Every list item checkoff.
- Every calendar event attendance.
- Every task regardless of purpose.
- Passive presence, page visits, or navigation.
- Reward shop purchases, future gem spending, or balance changes.
- Negative events such as missed chores, corrections, or reopened tasks.

Motivation should receive progress from a **Contribution Event** concept at the product level. A contribution event is not necessarily a new technical entity in the first slice, but the product model should think in those terms:
- What happened?
- Who contributed?
- Was it individual, shared, or family-level?
- What kind of contribution was it?
- Which active goals are eligible to count it?
- Was it automatically produced from a task or manually recognized by a parent?

This flow keeps Motivation downstream from household actions without making it the place where actions are managed.

## Task Relationship
The recommended direction is **Option C: Goals choose which Tasks count**, with a constrained version of Option B as the first implementation step.

### Why not Option A — Every Task counts
Every Task counting is simple, but it creates poor incentives:
- Children can farm easy tasks.
- Parents may hesitate to put mundane reminders into Tasks because they inflate Motivation.
- Zero-point or non-motivational responsibilities become emotionally loaded.
- The Motivation page becomes a task scoreboard instead of an encouragement surface.
- Shared family goals become volume contests.

### Why not pure Option B — Only selected Tasks count
Selected Tasks are safer, but if selection only lives on each task, parents must configure too much. It also makes goals less expressive because the task decides whether it is motivational even when different goals may need different criteria.

### Recommended — Option C with simple presets
Goals should define what counts. A goal can count:
- Specific selected tasks.
- Tasks with selected routine/category labels later.
- Tasks assigned to a specific Family Member.
- Shared Household tasks.
- Tasks completed within a time window.
- Parent-recognized Helpful Moments.
- A mix of the above, when deliberately configured.

The first implementation should not overbuild configuration. It can start with simple source presets:
- Selected tasks count toward this goal.
- Shared household task completions count toward this family goal.
- This Family Member completing selected tasks counts toward this individual goal.
- Parent-recognized moments count toward this goal.

The key rule: **a task can be complete without advancing Motivation.** Completion is still valid household work even when no Motivation goal listens to it.

## Family Goal Progress
Family goals should gain progress from household-level contribution patterns rather than individual competition.

Good family goal types:
- **Total contribution goal:** “Complete 20 helpful actions this week.”
- **Everyone participates goal:** “Everyone contributes once today.”
- **Shared work goal:** “Complete 5 shared household tasks.”
- **Routine rhythm goal:** “Complete 3 bedtime routine nights as a household.”
- **Helpful culture goal:** “Notice 5 helpful moments this week.”
- **Weekend reset goal:** “Finish the family reset before Sunday evening.”

Family goal types to avoid:
- “One child earns the most.”
- “Do as many tasks as possible.”
- “Avoid all mistakes.”
- “No one misses anything.”
- Goals whose only point is earning currency.

Progress calculation should match the goal type:

| Family goal type | Progress calculation | Child-facing copy |
| --- | --- | --- |
| Total contribution | Count eligible contribution events until target | “13 of 20 helpful actions” |
| Everyone participates | Count distinct participating Family Members | “3 of 4 helpers joined in” |
| Shared work | Count completed eligible Shared Household tasks | “2 of 5 shared jobs done” |
| Routine rhythm | Count completed routine days or routine occurrences | “3 bedtime wins” |
| Helpful culture | Count parent-recognized Helpful Moments | “4 helpful moments noticed” |
| Weekend reset | Count selected reset tasks or checklist groups | “Kitchen and laundry are done” |

Family goal progress should not expose raw per-child ranking. For an everyone-participates goal, Motivation can show inclusive checkmarks for participation, but it should avoid “Alex did 8, Sam did 1” as the primary family display.

## Individual Goal Progress
Individual goals should gain progress when a specific Family Member contributes to a goal-relevant behavior.

Good individual goal types:
- “Feed the pet 4 times.”
- “Complete bedtime routine 5 times.”
- “Help with dinner twice.”
- “Start homework setup without reminders 3 times.”
- “Do one helpful thing without being asked.”
- “Join one shared family job.”

Relationship between Tasks, Family Members, and Goals:
- A Family Member is the household participant being encouraged, not an account or security identity.
- A Task can be assigned to a Family Member, unassigned, or Shared Household.
- A goal can belong to one Family Member or to the family.
- A completed task can advance an individual goal only when both the actor/member and the task/source match that goal.
- Shared Household tasks should be able to credit the Family Member who participated if the household records who helped, but shared tasks should also be allowed to advance only the family goal when individual attribution would create friction.

For individual goals, parents should be able to choose whether progress comes from:
- The assigned Family Member completing their selected task.
- The Family Member being credited as a participant in a shared task.
- Parent recognition for initiative or helpfulness.
- Routine completions tied to that Family Member.

Individual progress should be private-by-tone even when visible on the shared Motivation page. Cards should have equal weight and should not sort by most progress.

## Helpful Moments
Helpful Moments should be a future Motivation concept because Tasks cannot represent all valuable household behavior.

Examples:
- Helped a sibling clean up.
- Took initiative without being asked.
- Comforted someone.
- Helped with dinner beyond assigned work.
- Noticed the recycling was full.
- Joined a shared task kindly.

Helpful Moments should affect progress **sometimes, by parent decision**.

Recommended behavior:
- A Helpful Moment can optionally count toward one or more active goals.
- The parent/caregiver chooses or accepts the default goal connection at recognition time.
- Some Helpful Moments should be recognition-only and not advance any count.
- Helpful Moments should never be a hidden automatic scoring rule.

Why not always:
- Always counting turns kindness into a farmable token source.
- Children may start optimizing for visible recognition instead of genuine help.
- Parents may feel pressure to record every nice behavior.

Why not never:
- It would miss initiative, cooperation, and emotional labor, which are central to the Motivation purpose.

Helpful Moments should be lightweight and warm: a short note, optional Family Member, optional goal contribution, and possibly a family-visible recognition feed later.

## Parent Recognition
Motivation should support manual progress sources, but manual recognition must be bounded.

Manual recognition is needed because:
- Not all contribution is task-shaped.
- Parents may want to credit effort, kindness, independence, or improvement.
- Younger children often need adult-observed praise rather than self-managed completion.
- Some households will not model every routine in Tasks.

Recommended manual recognition sources:
- Credit a Family Member for an individual goal step.
- Credit the family for a family goal contribution.
- Record a Helpful Moment without progress.
- Convert a Helpful Moment into goal progress.

Boundaries:
- No negative manual points.
- No manual deductions.
- No shame feed.
- No requirement to choose a reward value.
- No expectation that every contribution must be logged.

Parent recognition should be framed as “notice and encourage,” not “audit and pay.”

## Age Group Effects
The same underlying model can support ages 3-5, 6-8, and 9-12, but presentation and default goal design should differ.

### Ages 3-5
Best progress sources:
- Parent-recognized Helpful Moments.
- Very simple routine/task completions.
- Family participation goals.

Best presentation:
- Stars, stickers, checkmarks, helper path steps.
- Immediate celebration copy.
- Short goals with small targets.

Avoid:
- Numeric balances.
- Long-term saving.
- Complex task criteria.

### Ages 6-8
Best progress sources:
- Selected task completions.
- Weekly routine goals.
- Shared household contributions.
- Helpful Moments with parent approval.

Best presentation:
- Stars plus simple counts.
- Weekly goals.
- Family rewards or rituals.

Avoid:
- Leaderboards.
- Open-ended point farming.
- Long delayed rewards without intermediate progress.

### Ages 9-12
Best progress sources:
- Selected responsibilities.
- Initiative recognition.
- Shared work participation.
- Multi-step routines.

Best presentation:
- Progress bars, contribution counts, responsibility language.
- More autonomy and less childish visual language.
- Optional future links to Reward Economy only where the family deliberately enables it.

Avoid:
- Infantilizing stickers as the only mode.
- Arbitrary or opaque scoring.
- Surveillance-heavy reporting.

The model can stay unified: events, goals, members, progress units. Age affects defaults, labels, target sizes, and visual treatment.

## Progress Types
Progress units should be goal-specific and human-readable.

Recommended progress units:
- **Tasks completed** for selected task goals.
- **Shared tasks completed** for cooperation goals.
- **Routine completions** for repeated habits.
- **Days** for consistency goals.
- **Stars/checkmarks/steps** for child-facing progress.
- **Contributions** for mixed-source family goals.
- **Helpful Moments** for parent-recognized initiative.
- **Participants** for everyone-contributes goals.

What children should see:
- Simple progress language: “3 bedtime wins,” “2 more helpers needed,” “4 of 5 stars.”
- Warm next-step copy: “One more dinner helper moment to go.”
- Celebration when complete.
- No failure-heavy red states.
- No dense source audit.

What parents should see:
- The goal type.
- Which sources count.
- Which Family Member is linked, if individual.
- Whether progress came from Tasks, shared tasks, routines, or manual recognition.
- A small recent activity/history view later, not a punitive audit log.

Stars should be a presentation unit, not necessarily the underlying source. A star may represent one eligible contribution, one day, or one routine completion depending on the goal.

## Failure States
Missed goals should be handled encouragement-first.

Recommended behavior:
- **Time-bound weekly goals:** close the period, celebrate any progress, and optionally start a fresh goal.
- **Family goals with shared rewards:** parent decides whether to roll over, extend, or reset.
- **Individual goals:** avoid “failed” language; show “try again this week” or “we made progress.”
- **Routine goals:** allow partial success and restart without punishment.
- **Missed due dates:** Tasks may show overdue state, but Motivation should not punish progress.

Avoid:
- Negative points.
- Deducting progress.
- Public failure banners.
- Losing earned stars because of one missed day.
- Automatic shame notifications.

Default policy should be:
- Progress remains visible through the end of the period.
- At rollover, the next period starts fresh unless the parent extends the goal.
- Completed goals move to a small celebration/history area.
- Incomplete goals close with positive reflection, not penalty.

Parent decision is appropriate for family rewards because household context matters. A family may still choose movie night after strong effort, or extend the goal by two days, without the system implying inconsistency.

## Motivation vs Reward Economy
Motivation and Reward Economy should consume some of the same household events later, but they should not mean the same thing.

| Event | Motivation progress | Future Reward Economy balance | Rationale |
| --- | --- | --- | --- |
| Selected task completed | Sometimes | Sometimes | Depends on active goal and reward eligibility. |
| Any task completed | Usually no | Usually no | Avoid task farming and inflation. |
| Shared household task completed | Often for family goals | Rarely or configured | Encourages cooperation without automatically paying everyone. |
| Routine completed | Often for individual goals | Sometimes | Good for habits; currency should be intentional. |
| Helpful Moment | Sometimes by parent decision | Rarely | Recognition should not become automatic currency. |
| Parent recognition | Sometimes | Sometimes only if explicitly rewardable | Keeps praise separate from payment. |
| Goal completed | Yes, complete/celebrate | Maybe one configured reward event later | Shared rituals are not the same as spendable balances. |
| Reward redeemed/spent | No | Yes, decreases balance | Spending is economy behavior, not motivation progress. |
| Missed task | No negative progress | No negative balance by default | Encouragement-first. |
| Reopened task | May remove pending contribution if not final | Should not award until approved/final | Prevents false credit. |

Boundary rule:
- Motivation asks: “Did this help the child or family move toward an encouragement goal?”
- Reward Economy asks later: “Did this approved event create, spend, or adjust exchangeable value?”

A single event may feed both, but only when both domains have explicitly opted in.

## Long-Term Sustainability
Risks and mitigations:

### Gaming the system
Risk: children choose easy actions repeatedly.
Mitigations:
- Goals choose eligible sources.
- Avoid every-task counting.
- Use daily caps or one-credit-per-task-instance when needed.
- Prefer routine/participation goals over raw volume.

### Task farming
Risk: parents or children create trivial tasks to inflate progress.
Mitigations:
- Keep task completion separate from Motivation by default.
- Make Motivation eligibility explicit.
- Let family goals count selected task sets or shared categories, not arbitrary new tasks.

### Progress inflation
Risk: too many sources make goals meaningless.
Mitigations:
- Use one active family goal and a small number of individual goals.
- Keep targets modest and time-bound.
- Show progress units in plain language so inflation is visible.

### Goal inflation
Risk: parents create too many goals and children disengage.
Mitigations:
- Default to one active family goal.
- Limit individual active goals per child in early slices.
- Encourage weekly rhythm rather than many simultaneous goals.

### Parent workload
Risk: manual recognition becomes another chore.
Mitigations:
- Task-derived progress for selected goals.
- Helpful Moments as optional quick capture.
- Defaults that do not require source tuning for every task.
- Recognition-only moments that do not require deciding points or rewards.

### Fairness
Risk: older children, easier schedules, or more visible tasks dominate.
Mitigations:
- Use age-appropriate individual goals.
- Use participation goals that count each member once.
- Avoid leaderboards.
- Allow different targets per child without public comparison.

### Family dynamics
Risk: children compare progress or negotiate every action.
Mitigations:
- Lead with family goals.
- Show individual cards equally, not ranked.
- Keep rewards attached to goals and rituals, not every task.
- Preserve unrewarded responsibilities as normal.

## Risks
Motivation risks:
- Motivation becomes a second Tasks page if progress is too task-list-like.
- Motivation becomes fake if progress never moves or only moves through seeded numbers.
- Goals become confusing if source eligibility is invisible to parents.

Family dynamics risks:
- Sibling comparison if individual progress is presented competitively.
- Resentment if shared work credits one child more visibly than others.
- Younger-child disadvantage if raw counts dominate.
- Negotiation loops if every contribution has a reward implication.

Product risks:
- Blurred boundary with Gamification and Reward Economy.
- Parent configuration burden.
- Too many progress unit types too early.
- Overpromising automation before Tasks have recurrence, approval, or participant attribution.

Sustainability risks:
- Progress inflation from always-counting tasks.
- Helpful Moment overuse or underuse.
- Reward escalation if every goal needs a tangible prize.
- Loss of trust if children can self-complete rewardable tasks without any household agreement about verification.

## Open Questions
- Should the first progress slice update persisted current progress directly, or record source contributions for a minimal history view?
- Does a completed assigned task need an explicit “completed by” Family Member before it can advance individual progress, or is assigned owner sufficient for now?
- Should Shared Household tasks ask “who helped?” before they can count toward individual goals?
- How much source eligibility should parents configure in the first UI, if any?
- Should Helpful Moments launch before or after task-derived progress?
- Should missed/incomplete goal history be visible, hidden, or summarized only for parents?
- What language should replace “rewardLabel” if the product wants to avoid Reward Economy confusion?

## Recommended Direction
Use a goal-scoped contribution model.

Principles:
1. Motivation progress is generated by eligible contribution events, not by every activity.
2. Goals decide what counts.
3. Tasks are one input, not the entire Motivation system.
4. Helpful Moments and parent recognition are first-class future inputs for non-task helpfulness.
5. Family goals prioritize cooperation, shared routines, and inclusive participation.
6. Individual goals recognize age-appropriate growth without ranking siblings.
7. Failure states are reflective and encouraging, never punitive.
8. Reward Economy remains opt-in and separate from Motivation progress.

Recommended first source priority:
1. Selected task completion for selected active goals.
2. Shared Household task completion for family goals.
3. Manual parent progress/recognition for non-task contribution.
4. Helpful Moments as a named future capture model after the first progress path is understood.

## Recommended First Implementation Slice
The first implementation slice should be small and should not introduce Reward Economy, complex automation, recurrence, task approval, or a full goal builder.

Recommended slice: **Motivation Progress from Selected Task Completion**.

Scope:
- Allow active Motivation goals to be connected to a small set of existing Tasks or a simple eligibility preset.
- When an eligible task is completed, advance matching active goal progress.
- Support family goal progress from selected or shared tasks.
- Support individual goal progress when the goal's Family Member matches the task owner.
- Preserve current read-only display shape as much as possible.
- Add clear parent-facing source copy such as “Counts selected tasks” or “Counts shared household tasks.”

Keep out of scope:
- Helpful Moments UI.
- Reward balances, gems, tokens, shop, redemption, or avatar unlocks.
- Leaderboards.
- Negative points or penalties.
- Task recurrence and task approval.
- Complex goal editing.
- Multi-member task attribution unless already available.

If that slice feels too large, the smaller first slice should be manual parent progress only. However, selected task completion is more useful because Tasks already exist and the current missing product question is how progress actually moves.

## Next Prompt Context
HomeOps currently has persisted Family Goals, persisted Individual Goals, persisted Family Members, Tasks, a Home Motivation tile, and a Motivation page. Motivation progress is still stored as current read-model values and is not derived from household activity.

The next implementation prompt should ask for one narrow slice:
- Connect Motivation progress to selected eligible Task completions.
- Keep Motivation downstream of Tasks.
- Do not count every Task by default.
- Do not add Reward Economy, Helpful Moments, gems, tokens, shop, leaderboards, penalties, recurrence, approval, authentication, roles, or permissions.
- Keep the experience encouragement-first and avoid turning Motivation into a task board.
